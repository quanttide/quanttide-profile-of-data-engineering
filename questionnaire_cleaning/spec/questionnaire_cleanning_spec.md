# 中小制造业员工问卷数据清洗案例规范与业务蓝皮书 (questionnaire_cleaning_spec.md)

适用范围：本规范适用于传统中小制造业（以服装缝制加工为典型）的员工结构化问卷清洗任务。
核心目标：以业务意图为导向，将混乱、粗糙的原始问卷数据，清洗转化为能够支撑核心“有效产值元模型”分析的标准化数据集。
研发思想：理解情境 $\rightarrow$ 理解分析意图 $\rightarrow$ 在意图基础上构建契约（规范先于代码）。

---

## 1. 战略业务背景：极端混乱的数字化情境与行业痛点

纯计算机专业或初级数据工程师在面对此类项目时，往往容易陷入“为清洗而清洗”的代码泥潭，忽略了数据背后的极端情境。在传统制造业中，数据的业务价值高于技术美感。

### 1.1 中小制造业工厂的数字化现状
- **数字化水平极低**：缺乏昂贵的 ERP 或 MES 系统，生产记录、员工信息高度依赖纸质单据或极其粗糙的开源问卷工具。
- **生产情境极度混乱**：人员流动频繁、排班交叉、工序交错，跳转逻辑在采集时未被系统强制约束，充斥着噪声与“伪数据”。

### 1.2 核心业务：有效产值元模型
在制造业中，**产值**是企业生存的核心。所有的清洗动作，其终极目的都是为了构建并服务于以下核心公式：

$$\text{原始产值} = \text{员工数} \times \text{工序项} \times \text{工序产量}$$

> 📌 **关键业务认知**：在服装等制造业中，**“工序”是最小的分析单元**（如平缝、埋夹、锁边）。人和人产出多少只是结果，**产量必须精确到每一个工序项上产出多少**。

引入质量控制与损耗后，最终的**有效产值**计算公式为：

$$\text{有效产值} = \text{原始产值} \times (1 - \text{返工率})$$

整个问卷清洗的本质，就是围绕**如何准确构建这个产值模型**，以及组织清洗相关的辅助分析变量（如员工基本信息、组织归属、主观动机等）。

### 1.3 本项目的普适性行业价值
一个清洗预算几千块钱的项目，其背后解决的是无数中小企业花了几十万、几百万都没有搞清楚的**行业级大问题**——即**“如何面对实际上非常粗糙原始的数字化环境，提取出绝对准确的核心产值与效能数据”**。

我们搭建的数据云平台，恰恰是要在这种极端的、最恶劣的典型场景下顺畅工作。通过该案例提炼出的“业务契约化”方法论，对整个行业的数据清洗与工程场景具备极强的普适意义。

## 2. 研发思想三步法：非技术人员的协作基石

为了让团队中的新人及非技术岗（如项目经理、数据架构师）能够轻松理解并顺畅协作，平台研发与数据工程必须遵循以下三个核心步骤：

1. **理解情境**：深入传统制造业一线，接受其数字化表现形式极其粗糙、混乱的现实情境。
2. **理解分析意图**：明确一切清洗动作都不是为了炫技，而是为了服务于“有效产值公式”的构建，以及找出影响产值的管理/心理因素。
3. **在意图基础之上构建契约**：将业务团队与研发团队的共识沉淀为 **Codebook（数据契约）**。业务看懂指标，技术对照开发。

## 3. 数据契约定义 (Codebook 核心示例)

Codebook 是业务意图向技术落地的唯一权威协议。为了保持简洁清晰，此处仅抽取各业务单元的代表性字段进行技术与业务示例：

| 标准化变量名 | 原始来源 (列名/关键词) | 变量类型 | 值标签 / 映射规则 | 核心业务分析意图 (Why) |
| :--- | :--- | :--- | :--- | :--- |
| `time_spent_sec`| `所用时间` | 数值 | 正则提取“秒”，转为纯数字 | **辅助校验**：识别答题时间过短（如<30秒）的非认真样本，防止垃圾数据污染产值模型。 |
| `manager_name` | `2. 班组长姓名：` | 分类单选 | 模糊匹配：`1=陈亚梅`, `2=马秀华`... | **组织单元**：用于按“生产班组”聚合分析有效产值，识别高产能管理模式。 |
| `monthly_salary`| `7. 平均月工资是多少...`| 数值 | 连续数值（元/月） | **效能校验**：用于和计件工序计算出的理论产值进行对账，检验数据置信度。 |
| `satisfaction_task`| `分配给我的任务感到满意` | 数值量表 | 五级正向：`1=非常不同意` $\rightarrow$ `5=非常同意` | **隐性变量**：分析员工对任务分配的主观满意度是否会显著影响其生产返工率。 |
| `position_sewing`| `1. 目前的职位（可多选）`| 虚拟变量 | `1 = 选中车位缝纫工`, `0 = 未选中` | **生产力切片**：精准剥离核心生产人员（缝纫工）与后勤人员，准确锁定公式中的“员工数”。 |

## 4. 详细结构化清洗流程与代码映射规范

本节将生产情境中的“数字化混乱”转化为 Python 代码中的“高容错规则”，技术人员在编写脚本时必须严格执行。

### 4.1 缺失值与条件跳转的精细化拦截
传统问卷系统最严重的漏洞在于：容易将**“由于跳转逻辑导致的系统不适用缺失”**与**“用户展示了题目却主动跳过（漏填）”**混为一谈，这会导致后续分析有效产值的分母计算发生严重偏差。

#### 核心代码控制逻辑：
1. **显式跳过拦截**：代码中通过 `is_skipped_entry()` 函数统一拦截。任何观测值若为 `NaN`，或包含 `'跳过'`、`'(跳过)'` 字样（剔除空格后），均判定为跳过条目。
2. **互斥跳转多选题的特殊保护（重要）**：
   以“新员工加入原因”为例，只有在前置单选题选择“希望”时，系统才会展示该多选题；选择“不希望”的人，系统会自动在原始数据中填充 `(跳过)`。
   - **硬性清洗规则**：对于包含 `'(跳过)'` 标记的行，代码层面**必须将其清洗为 `NaN`（代表不适用），绝对不能填充为 `0`**。
   - **业务理由**：如果错误填充为 `0`，在后续统计“因为人手不够而希望新员工加入”的比例时，这些原本不适用、不需要回答的员工就会被错误地计入“分母”中，从而严重稀释、扭曲真正的员工动机分布。
3. **文本垃圾剔除**：在清洗社交网络（朋友姓名）以及开放式改进建议题（`improvement_suggestion`）时，若文本内容填写了 `'没有'`、`'空'`、`'无'`、`'(空)'` 等无意义词汇，代码层面直接强转为 `NaN`。

### 4.2 复杂多选题拆分的契约落地
在传统制造业的混乱采集线下，多选题原始单列文本中经常混合了系统奖励痕迹、自定义分隔符和开放文本后缀。
*例如原始单元格数据：`"车位缝纫工┋其他，请说明：底层打杂〖100元话费〗"`*

清洗代码必须严格按照以下四个步骤精密执行，以还原核心生产力标签：
1. **剥离系统痕迹**：利用正则 `re.sub(r'〖.*?〗', '', text)` 抹除类似话费验证等环境噪声。
2. **切分标准单元**：利用原始数据特有的 `┋` 分隔符进行 `.split('┋')` 切分。
3. **平整化选项文本**：利用正则 `re.sub(r'，请说明：.*', '', opt)` 抹除“请说明”及其后续输入，还原出纯净的 `车位缝纫工` 标准标签。
4. **派生虚拟变量（Dummy Variable）机制**：
   - 若该行判定为跳过/不适用，衍生出的所有虚拟变量一律赋 `NaN`。
   - 若当前有效样本命中对应选项，赋 `1`；未命中，赋 `0`。
5. **开放式文本单独提纯**：为了保留粗糙环境中员工自发填写的珍贵业务信息，单独衍生一个文本列（如 `position_other`）。当文本中包含“其他”时，将原始整段字符串完整保留，未选中且有效时赋空字符串 `''`。

### 4.3 包含式多选与常规衍生文本处理
对于“车型偏好”（如平车、埋夹车等）、“离职原因”等常规多选题，代码采用包含式检索逻辑（封装为 `create_multiple_choice_columns` 函数）：
- **常规映射**：若字段中包含 `工资太低`，则衍生变量 `quit_salary = 1`，否则在样本有效的前提下赋 `0`。
- **业务意义**：车型偏好和离职原因直接关联到产值公式中的“工序项配置”。将复杂的文本包含转化为二进制的 `0/1`，是为了让后续数据平台能够一眼看出到底是哪类工序设备偏好或人员离职摩擦在拖累总有效产值。

### 4.4 量表题的正反向计分规范
为了综合评估员工工作态度对有效产值（尤其是返工率）的隐性心理影响，量表题必须在数据清洗阶段统一“方向性”。
- **正向题**（如“对分配的任务感到满意”）：保持 `1 = 非常不同意` $\rightarrow$ `5 = 非常同意` 的常规映射。
- **反向题**（如“上级不体谅下属感受”、“工作任务太重”）：**必须在清洗阶段执行反向计分换算**（即通过代码将 `非常不同意` 映射为 `5`，`非常同意` 映射为 `1`）。
- **业务意义**：只有将所有心理量表题的得分方向全部清洗一致（高分统一代表积极正面，低分统一代表消极负面），后续业务团队在计算总得分或进行均值回归分析时，才可以直接相加。否则，正反混杂的原始分数会直接导致计算结果相互抵消，得出完全荒谬的业务结论。

## 5. 质量验证与平台契约审计标准

验证本规范是否合格的唯一标准是：**团队内的新人甚至是非技术岗，都能够利用本标准的输出，轻松核验清洗代码的确定性。**

### 1. 控制台“零带病”警告审计
清洗脚本（Data Pipeline）运行过程中，会实时触发列名匹配检查。
- 若日志中输出 `[警告: 未找到包含 'XXXX' 的列]` 或 `[警告: 列 'XXXX' 不存在]`，说明原始问卷由于数字化环境混乱，在前端修改了字段。
- **业务约束**：非技术人员（如项目经理）看到此警告，可直接判定数据契约被违背，**必须人工干预修正，严禁代码带病运行**。

### 2. 派生变量合并总量审计
多选题清洗完毕合并前，系统会输出 `[成功添加 XX 个任务偏好变量]`。
- **非技术核验法**：项目经理或 QA 无需翻阅具体的 Pandas 代码，直接核对该输出的变量数量是否等于 Codebook 契约中定义的虚拟变量总数。数量完全一致，即可闭环验证数据工程契约被完美履行。
