# GitHub 用户行为面板数据处理逻辑说明书（三期：作者归属 PR 采纳指标）

## 1. 模块目的与修订背景

本模块重建目标用户的 PR 提交及采纳指标。统计对象是“**目标用户提交、且在 GHTorrent 历史表中存在正式 `merged` 事件的 PR**”，而不是“目标用户执行的合并操作”。

旧模块 `pr_opened_merged_counter.py` 依赖 `filtered_history.csv` 判断合并状态。该中间表按照历史事件的 `actor_id` 筛选目标用户；但 `merged` 事件的 `actor_id` 通常是执行合并的维护者，而非 PR 作者。因此，目标用户向外部仓库提交、由样本外维护者合并的 PR 会被错误剔除。

本模块改为从未经 UID 过滤的 `pull_request_history.csv` 同时提取作者侧 `opened` 事件和全部 `merged` 事件，并通过 `pull_request_id` 关联。

---

## 2. 执行信息

**执行脚本**：`pr_authored_merged_counter.py`

**输入文件**：

* `uid.csv`：目标用户 ID 名单，必须含 `uid` 字段。
* `pull_request_history.csv`：全量 PR 历史表，字段顺序为 `id, pull_request_id, created_at, action, actor_id`。
* `pull_requests.csv`：PR 元数据表，用于获取 `base_repo_id` 和 `intra_branch`。
* `projects.csv`：项目表，用于获取目标仓库的 `owner_id`。
* `users.csv`：用户表，用于获取仓库所有者的 `type`，识别组织账号。

**输出文件**：

* `daily_pr_authored_merged.csv`：按 `(author_id, open_date)` 聚合的日度 PR 面板。
* `pr_reconciliation_report.txt`：样本数量、兜底数量、外键缺失和分类校验结果的核对报告。
* `pr_authored_merged_cache.sqlite`：可断点续跑的 SQLite 中间缓存，保存各阶段的事件、确权和分类关联结果。

**统一参数**：

* 分块大小：`chunksize = 200,000`。
* 样本窗口：`2011-01-01 <= open_date <= 2021-03-06`。
* 空值规则：将 GHTorrent 导出中的 `\N` 识别为空值；缺失关键外键时保留为 `unknown`，不得并入其他分类。
* 文件编码：输入按 UTF-8 读取，输出按 UTF-8 写入；日期输出格式为 `YYYY-MM-DD`。
* 缓存运行：首次运行创建 SQLite 缓存；中断后重新运行会复用已完成阶段。若源 CSV 或参数发生变化，使用 `python pr_authored_merged_counter.py --reset-cache` 删除缓存并全量重建。

## 3. 字段定义与统计口径

面板主键为 `(author_id, open_date)`。其中 `author_id` 是 PR 提交者，`open_date` 是 PR 的提交日，而非合并日。

| 字段名 | 定义与统计口径 |
| :--- | :--- |
| `author_id` | PR 作者 ID。优先采用 `action == 'opened'` 行的 `actor_id`；仅在缺失可匹配 opened 行时采用兜底规则。 |
| `open_date` | PR 提交日。优先采用 opened 事件日期；兜底 PR 采用该 PR 最早有效历史记录日期。 |
| `authored_pr_opened_count` | 该作者在该提交日纳入面板的 PR 数，包括按兜底规则识别的 PR。 |
| `merged_pr_count` | 当日提交的 PR 中，最终存在至少一条 `action == 'merged'` 历史事件的数量。一个 PR 只计一次。 |
| `self_pr_merged_count` | 被合并 PR 中，满足 `intra_branch == 1`，或目标仓库所有者 `owner_id == author_id` 的数量。 |
| `org_pr_merged_count` | 被合并 PR 中，目标仓库所有者在 `users.csv` 中的 `type == 'ORG'` 的数量。该类单列，不并入外部采纳。 |
| `external_pr_merged_count` | 被合并 PR 中，目标仓库属于其他非组织个人账号的数量。 |
| `unknown_pr_merged_count` | 被合并 PR 中，因 PR 元数据、目标仓库所有者或所有者类型无法确权而不能分类的数量。该类单列输出。 |
| `merge_lag_mean` | 被合并 PR 的平均合并时滞，计算为 `merge_date - open_date` 的天数均值；当日无 merged PR 时为空。 |

分类恒等式为：

```text
self_pr_merged_count
+ org_pr_merged_count
+ external_pr_merged_count
+ unknown_pr_merged_count
= merged_pr_count
```

脚本在写出每一行前使用 `assert` 校验该恒等式。

## 4. 全链路算法与数据流向

### 4.1 作者侧 PR 与合并事件提取

1. 分块扫描全量 `pull_request_history.csv`，不使用 `filtered_history.csv`。Pandas 在每个数据块内完成 `action`、目标用户和日期窗口的向量化筛选；筛选结果批量写入 SQLite。
2. 对 `action == 'opened'` 的记录，仅保留 `actor_id` 位于 `uid.csv`、且事件日期落在研究窗口内的 PR。以 `pull_request_id` 为键，保留最早的有效 opened 记录：

   ```text
   authored_opens[pr_id] = (author_id, open_date)
   ```

3. 对 `action == 'merged'` 的记录，不对 `actor_id` 施加任何筛选。以 `pull_request_id` 为键保留最早的 merged 日期：

   ```text
   merged_dates[pr_id] = merge_date
   ```

   若同一 PR 存在重复 merged 记录（例如 reopen 后再次合并），仍只作为一个最终被合并的 PR 计数。

### 4.2 缺失 opened 行的兜底识别

对存在 `merged` 事件、但不能与 `authored_opens` 匹配的 PR，脚本再次分块扫描原始 `pull_request_history.csv`：

1. 按 `pr_id` 查找该 PR 全部历史记录中时间最早的一条有效记录。
2. 仅当该最早记录的 `actor_id` 位于 `uid.csv`、且其日期处于研究窗口内时，使用它作为作者和提交日代理：

   ```text
   fallback_author[pr_id] = (earliest_actor_id, earliest_event_date)
   ```

3. 该规则使用的是**最早历史记录**，不是最早 merged 记录；merged 日期仍专门用于识别采纳状态和计算合并时滞。
4. 最早记录不属于目标用户、日期超出窗口或关键字段缺失时，该 PR 不纳入作者日度面板。

### 4.3 作者—合并关联与 PR 元数据挂接

将正常识别的 `authored_opens` 与符合兜底条件的 `fallback_author` 合并为 `authored_prs`。随后 SQLite 以 `pr_id` 与 `merged_dates` 做内连接，得到“目标用户提交且最终被合并”的 PR 集合。中间表均持久化于缓存文件中，因此后续步骤失败时无需重扫已完成的全量 CSV。

脚本分块读取 `pull_requests.csv`，仅对该集合挂接：

* `intra_branch`；
* `base_repo_id`，即 PR 合并的目标仓库。

分类和所有权均以 `base_repo_id` 为准，不以 head 仓库、合并人身份或 merged 行 `actor_id` 为准。

### 4.4 目标仓库确权与四分类

1. 对 `intra_branch != 1` 的 PR，分块扫描 `projects.csv`，建立：

   ```text
   repo_to_owner[base_repo_id] = owner_id
   ```

2. 对已识别的 `owner_id`，分块扫描 `users.csv`，建立：

   ```text
   owner_to_type[owner_id] = users.type
   ```

3. 依照以下优先级分类：

   ```text
   intra_branch == 1                         -> self
   owner_id == author_id                     -> self
   users.type == 'ORG'                       -> org
   owner 已确权且 type 非 'ORG'               -> external
   PR / project / owner / type 外键缺失       -> unknown
   ```

`unknown` 不得强制并入 `external`，以保留数据缺失的可审计性。

### 4.5 日度面板聚合

对全部 `authored_prs` 按 `(author_id, open_date)` 聚合 `authored_pr_opened_count`。对其中已合并的 PR，在相同主键下累加 `merged_pr_count` 和四类分类计数；计算每条 PR 的：

```text
merge_lag_days = (merge_date - open_date).days
```

随后按相同主键计算 `merge_lag_mean`。因此，某 PR 即使在后续日期才合并，仍归因于它的提交日；该设计衡量的是用户当日提交 PR 的后续采纳结果。

## 5. 数据验证与核对报告

`pr_reconciliation_report.txt` 至少披露以下结果：

* 目标用户数量、历史表两次扫描行数；
* 窗口内正常识别的目标用户 opened PR 数；
* 全量存在 merged 事件的 PR 数；
* 最终进入面板的“作者提交且被合并”PR 数；
* 兜底识别的已合并 PR 数及其占最终样本的比例；
* 有 merged 但未匹配到目标用户 opened 的兜底候选 PR 数；
* 缺失 `pull_requests` 元数据的 PR 数、未确权目标仓库数、缺失 `users.type` 的 owner 数；
* self、org、external、unknown 四类的总体数量；
* 新版可比第三方采纳（`org + external + unknown`）与二期模块旧 external 基准 66,614 的差异及核验结论；
* 合并日期早于提交日期的异常记录数，供后续审计；
* 每一行分类恒等式断言通过的结果。

## 6. 数据边界与解释限制

1. **可观测采纳下界**：GHTorrent 的 `merged` 事件主要捕捉通过 GitHub 界面或 API 合并动作完成的 PR。维护者若采用 rebase、界面外 squash 或 cherry-pick 等方式吸收贡献，可能不会留下 merged 事件。因此 `merged_pr_count` 及其四类拆分是实际采纳数量的可观测下界。
2. **右删失**：样本以提交日为窗口，截止日期为 2021-03-06。接近截止日提交的 PR 后续观察时间较短，其最终被合并的机会可能尚未完全实现。解释合并率或时滞时应考虑该右删失问题。
3. **兜底记录的代理属性**：缺失 opened 行时，最早历史记录的 actor 和日期并不等同于严格可观测的 PR 创建事件。该部分已单独在核对报告中披露数量及占比，研究中应将其视为代理识别，并可作为稳健性分析中的敏感性样本处理。

## 7. 新旧口径结果对比与核验

本节使用二期模块旧面板 `daily_pr_opened&merged_count.csv` 与本模块运行生成的 `pr_reconciliation_report.txt` 进行总体核验。旧面板中的 `external_pull_request_merged_count` 未将组织仓库单列；新面板则将组织账号拥有的目标仓库拆分为 `org_pr_merged_count`。因此，新旧结果应按可比的汇总口径解释。

### 7.1 本次运行的总体结果

| 指标 | 旧口径 | 新口径 | 差异 |
| :--- | ---: | ---: | ---: |
| 目标用户 opened PR 数 | 857,549 | 856,773 | -776（-0.09%） |
| 全量存在 merged 事件的 PR 数 | — | 53,894,653 | — |
| 最终被合并 PR 数 | 374,517 | 612,414 | +237,897（+63.52%） |
| 自有仓库合并（self） | 307,903 | 406,980 | +99,077（+32.18%） |
| 组织仓库合并（org） | 未单列 | 167,352 | 新增单列 |
| 外部个人仓库合并（external） | 66,614 | 38,082 | 不宜直接比较 |
| 无法确权（unknown） | 0 | 0 | 一致 |

新口径中，缺失正常 opened 记录而通过最早有效历史记录兜底识别、并最终进入已合并作者 PR 样本的数量为 64,989 条，占新口径最终已合并作者 PR 的 10.61%。

本次运行未出现已合并作者 PR 的 `pull_requests` 元数据缺失、目标仓库无法确权或已确权 owner 缺失 `users.type` 的情况，三项数量均为 0；`merge_date` 早于 `open_date` 的异常 PR 为 5 条，保留供后续审计。

### 7.2 可比的第三方采纳口径

旧模块的 external 类别没有将组织仓库单列；新模块中，组织仓库和其他个人仓库均属于“非作者本人仓库的第三方采纳”，但分别保存为 org 与 external。因此，不应将旧版的 66,614 条与新版纯个人 external 的 38,082 条直接比较。

应使用以下可比汇总：

```text
新版第三方采纳 = org_pr_merged_count
              + external_pr_merged_count
              + unknown_pr_merged_count
              = 167,352 + 38,082 + 0
              = 205,434
```

相对旧模块 external 总数 66,614 条，新口径的第三方采纳增加 138,820 条（+208.39%，为旧值的约 3.08 倍）。这一增量与重建预期一致：不再因 merged 事件由样本外维护者执行而系统性遗漏作者提交后被采纳的 PR。

### 7.3 核验结论

1. 两版目标用户 opened PR 总量仅相差 776 条，表明作者侧基础样本识别总体一致；微小差异可能来自输入过滤、缺失值清理及异常行跳过规则不同。
2. 新版最终被合并 PR 总量显著增加，且新增组织仓库类别，说明重建成功恢复了旧口径遗漏的第三方采纳记录。
3. 本次运行中 PR 元数据、目标仓库 owner 与 owner type 的外键缺失均为 0，四类分类计数与 merged 总数的恒等式断言通过。
